# 海克斯象棋（Hextech Chess）开发任务书

> 这是给 Codex 的完整开发提示词。请通读全文后，按"开发里程碑"分阶段实现，每完成一个阶段就提交一次 commit。

## 一、项目愿景

开发一款名为 **海克斯象棋（Hextech Chess）** 的网页游戏。它以**中国象棋**为规则骨架，融入受《英雄联盟》海克斯大乱斗启发的**符文系统**：玩家在对局中获得可改变棋子能力或棋盘规则的"海克斯符文"，让每一局都有新鲜变化。

核心设计原则（必须贯穿始终）：
1. **公平性优先**：随机性必须是"对称的随机"。双方面对相同的符文池、相同的能量经济、相同的选择机会。没有任何一方能靠运气获得不可逆的碾压优势。
2. **娱乐性**：符文之间能产生 combo，每局体验不同，有"开盲盒"的爽感。
3. **UI 一流**：海克斯科技美学（深蓝、青绿、金色、发光符文、六边形元素），动画流畅，桌面和移动端都好用。

## 二、技术栈（请严格采用）

- **框架**：React 18 + TypeScript + Vite
- **样式**：Tailwind CSS
- **动画**：Framer Motion（棋子移动、符文激活特效）
- **状态管理**：Zustand（轻量、适合游戏状态）
- **图标/字体**：六边形 SVG 自绘 + Google Fonts（标题用 Orbitron 或 Rajdhani 一类科技感字体）
- **测试**：Vitest（象棋规则引擎必须有单元测试）
- **代码规范**：ESLint + Prettier，全程严格 TypeScript（开启 strict 模式）
- **部署**：GitHub Pages + GitHub Actions 自动构建；保留自定义域名的 CNAME 配置

⚠️ 重要：第一阶段做成**纯前端静态应用**（对战 AI + 本地双人热座），不依赖后端，这样能直接部署到 GitHub Pages。在线联机作为可选的后期阶段处理。

## 三、象棋规则引擎（基础层）

实现标准中国象棋规则，封装成一个**纯函数、无 UI 依赖、可独立测试**的引擎模块（`/src/engine/`）：

- 棋盘 9×10，含楚河汉界
- 棋子：将/帅、士/仕、象/相、马/傌、车/俥、炮/砲、卒/兵
- 完整走子合法性：马蹩腿、象眼、炮隔山打、将帅不能照面、过河兵规则等
- 判定：将军、将死、困毙、和棋
- 提供 API：`getLegalMoves(state, square)`、`applyMove(state, move)`、`getGameStatus(state)`
- **所有规则必须有 Vitest 单元测试覆盖**（包括边界情况）

符文系统会通过"修改器（modifier）"机制挂载到引擎上，所以引擎要设计成可扩展：走子规则、吃子结算、回合开始/结束都要预留 hook，让符文能注入逻辑，而不是把符文逻辑硬编码进基础规则。

## 四、海克斯符文系统（核心创新层）

### 4.1 能量经济（保证公平的基础）

- 双方各有一条**能量条（Hex Energy）**，从 0 开始
- 每个回合开始时，当前行动方 +1 能量（上限例如 10）
- 主动符文需要消耗能量激活；被动符文在选取时一次性生效
- 双方能量增长完全对称 → 这是公平性的地基

### 4.2 符文的获取方式（对称随机，公平的核心）

采用 **镜像 Ban/Pick + 蛇形征召** 的开局阶段：

1. 系统从符文池中随机抽出 **N 张候选符文**（双方看到的是同一组候选，这就是"对称随机"）
2. 双方按蛇形顺序轮流选取：先手选 1 → 后手选 2 → 先手选 2 → …直到各自凑满符文位（例如各 3 个）
3. 每张符文有**稀有度与点数成本**（白银/黄金/棱彩），双方的总点数预算相同，防止有人堆叠超模符文

> 可选增强：对局中每隔若干回合，再触发一次"小符文刷新"，双方同时从相同候选里各选一张，持续带来变化但仍对称。

### 4.3 符文分类与示例（请实现至少 18 张，分布在各类）

**A. 棋子强化（被动）**
- 「机械炮台」：你的炮第一次开火无需炮架
- 「不羁之马」：你的马免疫蹩马腿
- 「逆行卒」：你的过河兵可以后退一格
- 「钢铁堡垒」：你方将/帅周围一格内的己方棋子获得 1 层护盾（首次被吃免疫一次）

**B. 主动技能（消耗能量）**
- 「海克斯传送」（4 能量）：将一枚己方棋子瞬移到棋盘上任意空格（不能直接将军，避免破坏平衡）
- 「时空错位」（3 能量）：交换两枚相邻己方棋子的位置
- 「符文护盾」（2 能量）：本回合给一枚棋子加 1 次免死护盾
- 「亡者归来」（6 能量）：复活一枚被吃掉的己方卒/兵，放回我方底线

**C. 棋盘改造（被动/一次性）**
- 「海克斯传送门」：在棋盘上生成一对传送门格子，棋子踩上去可在两门间穿梭
- 「裂隙」：移除楚河汉界对某一列象/相过河的限制（仅一列）

**D. 光环（被动）**
- 「集结号令」：每吃掉一枚敌方棋子，+1 额外能量
- 「连锁反应」：当你用车吃子时，可顺势再走该车一格（不吃子）

**实现要求**：
- 每张符文是一个数据驱动的对象（id、名称、描述、稀有度、点数、类型、触发时机、效果函数）
- 所有效果通过引擎的 hook 注入，**严禁**写成无法关闭的全局补丁
- 每张符文配一段简短的"海克斯风味文案"
- 必须做**平衡性数值表**，把每张符文的预估强度标注出来，方便后续调整

### 4.4 公平性自检清单（请在代码里以注释或文档体现）

- [ ] 双方面对的随机候选池完全相同
- [ ] 双方符文点数预算相同
- [ ] 能量增长规则对双方一致
- [ ] 没有任何符文能产生"一回合即胜且对手无法应对"的效果
- [ ] 主动符文不能直接造成将死（防止跳过对抗）

## 五、对手与游戏模式

- **本地双人（热座）**：同一设备轮流操作，第一阶段必做
- **人机对战**：象棋 AI 用 minimax + alpha-beta 剪枝实现基础走子评估；符文带来的状态变化需纳入评估（先做能正常对弈的中等难度，难度可调：简单/普通/困难）
- AI 也应该会使用符文（按简单启发式：能量够且符文有正收益就考虑使用）

## 六、UI / UX 设计要求

整体走**海克斯科技（Hextech）美学**：

- **配色**：背景深蓝近黑（#0A1428 一类），主色青蓝（#0AC8B9 / #0397AB），点缀金色（#C8AA6E / #F0E6D2），符文发光用霓虹高光
- **元素**：六边形纹理、电路线条、发光描边、卡牌质感的符文卡
- **棋盘**：清晰可读，棋子用中文字 + 发光边框；选中棋子时高亮可走位置（用半透明圆点）
- **动画**（Framer Motion）：棋子滑动移动、吃子时的粒子/碎裂特效、符文激活时的光效、能量条充能动画
- **界面布局**：
  - 顶部：双方信息（头像、能量条、已拥有符文图标）
  - 中央：棋盘
  - 侧边/底部：当前可用符文卡（hover 显示详情）、能量、走子历史、悔棋/认输按钮
- **开局符文征召界面**：要做得有仪式感，像抽卡/选符文，卡牌翻转、稀有度光效
- **响应式**：桌面横向布局，移动端纵向自适应，触控友好
- **音效**（可选但加分）：走子、吃子、符文激活、胜利音效，提供静音开关

请参考"卡牌对战游戏 + 科技 UI"的成熟设计语言，**避免千篇一律的 AI 生成感**：要有明确的视觉层级、留白、动效节奏。

### 6.1 图像素材生成（由你用 image2 自行生成）

本项目**不提供任何外部图片资源**。所有图像素材一律由你（Codex）调用 **image2** 工具自行生成，不要使用占位图、第三方图库链接或外链 CDN 图片。

需要生成的素材至少包括：
- 游戏 Logo / 标题图（"海克斯象棋"科技字标）
- 网站 favicon / 应用图标
- 各符文卡牌的插画（每张符文一张配图，风格统一为海克斯科技魔法风）
- 符文稀有度边框 / 卡牌底纹（白银、黄金、棱彩三档）
- 双方玩家头像 / 阵营标识（蓝方、红方）
- 棋盘背景纹理、能量条素材、胜利 / 失败结算画面背景
- 主菜单 / 征召界面的氛围背景图

生成与使用规范：
- 所有生成的图片统一存放在 `/public/assets/` 下，按类型分子目录（如 `assets/runes/`、`assets/ui/`、`assets/branding/`）
- 全部素材风格必须统一：海克斯科技美学，深蓝 + 青绿 + 金色，发光、六边形、电路魔法元素，符合第六节的配色与视觉语言
- 矢量类素材（图标、边框、棋子发光描边、简单几何）优先用代码 / SVG 自绘，位图插画类（符文插画、背景、Logo）用 image2 生成
- 生成的图片要做好尺寸与体积优化（适当压缩，避免拖慢加载）
- 所有素材随仓库一起提交，确保 GitHub Pages 部署后能正常加载，**不依赖任何运行时外部图片请求**

## 九、部署与交付要求

- 配置 `.github/workflows/deploy.yml`，push 到 main 分支即自动构建并发布到 GitHub Pages
- 所有由 image2 生成的图像素材必须随仓库一起提交（位于 `/public/assets/`），部署后不依赖任何外部图片请求
- `vite.config.ts` 的 `base` 要正确（用自定义域名时一般为 `/`，用 `用户名.github.io/仓库名` 时为 `/仓库名/`），并在 README 写清两种情况怎么改
- 在 `public/` 放一个 `CNAME` 文件占位，并在 README 说明：购买域名后，把域名填入 CNAME，并在域名服务商处配置 DNS（CNAME 指向 `用户名.github.io`，或用 A 记录指向 GitHub Pages 的 IP）
- README 包含：项目介绍、本地运行命令、规则与符文说明、部署步骤、域名绑定步骤
